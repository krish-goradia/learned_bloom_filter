system design

Architecture

Standard Bloom Filter

Baseline — all bad training URLs inserted directly.

Query URL → Bloom Filter (stores ALL bad train URLs) → bad / good Guarantee: zero false negatives Cost: memory scales with |bad URLs|

Learned Bloom Filter

Model handles bulk classification; BF patches training errors only.

Query URL → ML Oracle (LR + HashingVectorizer) prob ≥ threshold? YES → flag bad ✓ NO → Backup BF (stores train FNs only) found? → bad / good

C++ Bloom Filter

Built from scratch, exposed to Python via pybind11.

Hash: xxHash64 + double hashing idx = (h1 + i·h2) % m Storage: vector<uint64_t> bit array cache-friendly, 64-bit aligned Sizing: m = -(n·ln(fpr)) / ln(2)² k = (m/n)·ln(2)

ML Oracle

Stateless vectorizer — no vocabulary storage.

Vectorizer: HashingVectorizer n_features: 1024 / 2048 / 4096 / 16384 alternate_sign: False Model: LogisticRegression max_iter: 1000 input: raw URL string output: P(bad URL)

Findings

Memory · Crossover

Learned BF achieves 16–32% memory reduction at FPR ≥ 0.01. At FPR = 0.005, learned BF (0.072 MB) exceeds standard BF (0.070 MB) by 3.6% — feature space inflation at n_features = 4096 erases gains. The crossover is driven by model weight cost (n_features × 64 bits) dominating as larger feature spaces are needed for tighter precision.

Latency · Overhead

Consistent 1.69–1.87× latency overhead across all FPR targets (~185–205 ns vs ~109 ns). The overhead is stable because model inference and BF lookup costs are largely independent of the FPR target — it is structural, not tunable.

Latency · Breakdown

Model (~91–99 ns, ~49%) and backup BF (~104–112 ns, ~54%) contribute roughly equally to total latency. Neither dominates — optimizing either component alone recovers at most half the overhead. Standard BF cost sits at approximately the model-only level.

Throughput

Standard BF sustains ~9.1–9.2M queries/s vs learned BF's ~4.9–5.4M queries/s. The ~1.7× throughput gap directly mirrors the latency overhead — the sequential model + BF pipeline approximately halves effective throughput relative to a single BF lookup.

FNR · Oracle Quality

FNR drops 38–44% as n_features scales from 1024 to 16384 at any fixed threshold. More hashing buckets reduce collisions, improving oracle generalization. However, this gain comes at a linear model memory cost — creating a direct FNR–memory tradeoff independent of the BF component.

FNR · Responsibility

System FNR reflects model generalization error, not a BF failure. The backup BF guarantees zero false negatives on its own inserted elements by construction — it cannot catch bad URLs the model misses at test time, as BFs memorize rather than generalize. FNR is entirely the oracle's responsibility.

Limitations

  • L1
    Worst-case evaluation by design. The random 70/30 split produces disjoint train/test bad URL sets — zero overlap by construction. This means the backup BF cannot recover any test-time false negatives, representing the worst case for the backup BF. This is intentional: it isolates model generalization as the sole performance driver. Simulating overlap would constitute data leakage, conflating memorization with generalization. In realistic deployment where known malicious URLs are repeatedly queried, the backup BF would meaningfully reduce system FNR. Our evaluation reports a conservative lower bound.
  • L2
    Weak oracle. Logistic Regression on hashed character features is a deliberately simple baseline. A character n-gram TF-IDF model or small MLP would produce fewer training false negatives, shrink the backup BF further, and reduce system FNR. Oracle strength is the primary lever for improving all downstream metrics.
  • L3
    Memory accounting asymmetry. Standard BF memory includes only BF bits. Learned BF memory includes model weights + BF bits — this is the correct total system cost comparison, but the components are structurally different. Results should be read as total deployment memory, not BF-to-BF.
  • L4
    Moderate class imbalance. The dataset is 80% benign / 20% malicious with no resampling applied. Class-weighted training or oversampling could improve oracle quality, particularly FNR at high thresholds.
