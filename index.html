system design

Architecture

Standard Bloom Filter

Baseline — all good (in-set) training URLs inserted directly.

Query URL → Bloom Filter (stores ALL good train URLs) → pass / block Guarantee: zero false negatives FP: bad URL wrongly passed (bounded by FPR) Cost: memory ∝ |good URLs| × bits-per-element

Learned Bloom Filter

Model handles bulk classification; backup BF patches training FNs only.

Query URL → ML Oracle (LR + Vectorizer) P(good) ≥ threshold? YES → pass ✓ NO → Backup BF (stores training FNs only) found? → pass / block

C++ Bloom Filter

Built from scratch in C++17, exposed to Python via pybind11.

Hash: xxHash64 + double hashing idx = (h1 + i·h2) % m, i ∈ 0..k Storage: vector<uint64_t> bit array 64-bit aligned, cache-friendly Sizing: m = -(n·ln(fpr)) / ln(2)² k = (m/n)·ln(2)

Problem Framing

Membership testing — not binary classification.

good URL (label=1) → in-set member bad URL (label=0) → non-member Train: ALL good + 80% bad Test: ALL good + 20% bad (disjoint) FNR = 0 by construction: all good URLs seen at train time → backup BF catches every model FN → system never blocks a member
vectorisation strategy

Three Vectorisers, One Story

The project evolved through three vectorisation strategies, each trading model quality against latency differently. The progression from word tokens → char trigrams → fused C++ kernel tells the story of closing the gap between a research prototype and a deployable system.

Step 01 · Python
HashingVectorizer
Word Tokens (Default)

scikit-learn's default word-level tokeniser splits URLs on whitespace and punctuation, producing ~3–8 word tokens per URL. Each token maps to a hashed feature bucket. Fast to run but discards all subword structure — path components, subdomains, and query strings are split into fragments that lose their malicious signal context.

HashingVectorizer( analyzer="word", n_features=n_features ) avg NNZ per URL: ~6 batch latency: ~7.5 microseconds/query

Step 02 · Python
HashingVectorizer
Char Trigrams

Switching to analyzer='char', ngram_range=(3,3) captures subword patterns like .ph, /ad, :// that word tokens miss entirely. A URL of length L produces L−2 trigrams, averaging ~160 NNZ per row — 26× denser than word tokens. Model quality improves substantially, but latency explodes because scikit-learn's sparse matrix pipeline is bottlenecked on CSR construction and batch numpy operations.

Step 03 · C++17
Fused Rolling Hash
Char Trigrams

The C++ kernel eliminates Python overhead entirely. Vectorisation and scoring are fused into a single pass over the URL string — no CSR matrix is built, no intermediate data is allocated. A rolling hash computes each trigram in O(1). Model weights exported from sklearn are stored in a double[] array; the logit accumulates inline. All latency figures are measured from Python via time.perf_counter_ns(), wrapping a single batched pybind11 call, with warmup runs before timing and the mean taken over 10 trials divided by batch size — consistent methodology across all variants.

important · latency note

All latency figures — Python and C++ alike — are mean amortised per-query costs measured consistently from Python using time.perf_counter_ns(), wrapping a single batched call per variant with 3 untimed warmup runs followed by 10 timed trials, divided by batch size. This methodology captures the full Python↔C++ crossing overhead on both sides and reflects steady-state throughput under batch load, not single-query online latency. The 54× gap between C++ and Python char trigrams reflects the architectural difference — eliminating CSR construction, intermediate allocation, and generic sklearn call.

Benchmarks

Fig 1 — Memory Usage: All Variants vs Standard BF
Grouped bars show best-config total memory (model weights + backup BF bits) for each variant at four target FPR values. The dashed red line is the standard BF reference. Percentage labels show memory reduction vs standard BF.
Analysis. The C++ trigram variant achieves the largest savings at every FPR target — up to 72% at FPR = 0.1 and 47% at FPR = 0.01. The word-default vectoriser consistently underperforms char trigrams because word tokens produce fewer, coarser features, leaving the model less discriminative and pushing more URLs into the backup BF. At FPR = 0.005, all three learned variants still beat the standard BF (30–16% savings), contradicting a simple narrative of a hard crossover — the savings narrow but do not vanish. The char trigram Python variant sits between the C++ implementation and the word default, confirming that feature quality (not just vectoriser speed) drives memory efficiency.
Fig 3 — Threshold Tradeoff (C++ Trigram, Target FPR = 0.01)
Left: classification threshold vs achieved system FPR across all n_features values. Dashed line marks the 0.01 target. Right: same threshold axis vs training FN count, which directly controls backup BF size and memory.
Analysis. The two panels expose a fundamental tension in the learned BF design. Lowering the threshold passes more URLs through the model without querying the BF — reducing FPR — but simultaneously classifies fewer good URLs as "definitely good", increasing the training FN count and therefore the backup BF size. The right panel shows fn_count growing steeply above threshold = 0.90, explaining why high-threshold configs dominate memory despite their better FPR performance. The n_features dimension shows that larger feature spaces push the FPR curve down and left (better FPR at the same threshold) but the fn_count curves overlap closely — more features help FPR more than they help backup BF size. This means the memory savings from better oracle quality accrue primarily through being able to use a lower threshold to hit the same FPR target.
Fig 4 — Memory Reduction Heatmap: Threshold × n_features (C++ Trigram)
Each cell shows the percentage memory reduction vs standard BF for the best backup_fpr at that threshold / n_features combination. Grey cells (—) indicate no valid configuration exists that meets the target FPR constraint. Left: target FPR = 0.01. Right: target FPR = 0.05.
Analysis. The heatmap makes the operating regime immediately visible. At FPR = 0.01 (left), valid configs only appear at high thresholds (≥ 0.90) — the model needs extreme confidence to suppress bad URL leakage below 1%. Even within valid configs, memory reduction varies significantly: threshold = 0.99 at n_features = 4096 achieves ~31%, while lower feature counts at the same threshold achieve less because the weaker oracle pushes more URLs into the backup BF. At FPR = 0.05 (right), the valid region expands substantially — configs with threshold ≥ 0.70 become viable and memory savings reach 60%+. The takeaway is that the learned BF's operating regime is highly sensitive to the FPR target: relaxing from 0.01 to 0.05 roughly doubles the achievable memory savings.
Fig 5 — Memory Frontier: All Variants Across FPR Targets
Best-config memory for each variant plotted against target FPR on a log scale. The shaded region below the C++ trigram line and above the standard BF line represents the memory savings achieved by the learned approach.
Analysis. This is the summary view. All three learned variants beat the standard BF at every FPR target tested — there is no crossover in this dataset with this problem framing. The gap between variants widens as FPR relaxes: at FPR = 0.1, char trigrams and C++ trigrams nearly converge (both ~0.054 MB) while the word-default variant lags at 0.083 MB. At tight FPR targets (0.005), the spread narrows because all variants need large feature spaces and high thresholds, both of which inflate memory toward the standard BF level. The log x-axis reveals that memory scales roughly linearly with log(1/FPR) — each order of magnitude tighter in FPR requirement roughly doubles total system memory across all variants.

analysis

Findings

Memory · Feature Quality

Char trigrams consistently produce better memory savings than word tokens across all FPR targets — the gap is largest at relaxed targets (72% vs 58% at FPR = 0.1) and narrows at tight targets. Trigrams capture subword patterns like .php, /admin/, and %20 that word tokens split apart, giving the model stronger discriminative signal and reducing training FNs that inflate the backup BF.

Memory · Threshold Tradeoff

Every FPR target has a minimum achievable threshold below which no valid config exists. Higher thresholds lower FPR but raise fn_count, directly inflating backup BF size and eroding memory savings. At FPR = 0.005, the threshold must reach 0.99 for the C++ variant — pushing fn_count above 150K. At FPR = 0.1, threshold = 0.50 suffices with fn_count well under 50K.

Latency · C++ vs Python

The C++ implementation achieves ~0.5 microseconds mean amortised per-query cost, compared to ~26 microseconds for Python char trigrams — a 54× difference. Both numbers are measured identically: Python time.perf_counter_ns() wrapping a single batched call, 3 warmup runs, mean of 10 trials, divided by batch size. The gap is architectural — eliminating CSR construction, intermediate allocation, and generic sklearn dispatch — not a difference in measurement methodology.

Latency · Structural Cost

The C++ learned BF at ~0.5 microseconds total is 2.5× slower than the standard BF at ~0.2 microseconds . This overhead is structural — the model scoring pass is an unavoidable addition to the BF lookup. The standard BF will always win on latency; the learned BF trades latency for memory.

FNR · Zero by Construction

System FNR = 0.000 across all 420 configurations (140 per variant). This is not a model quality result — it follows from the experimental design. All good URLs appear in both train and test sets. The backup BF sees every member the model misclassified at training time, and since those same members reappear at test time, recovery is guaranteed. A weaker model simply routes more URLs through the backup BF at higher memory cost.

n_features · Optimal Point

Across all three variants and all FPR targets, n_features = 4096 dominates best-config selections. Larger feature spaces (16384) reduce hash collisions and improve oracle quality, but the linear growth in model weight memory (n_features × 8 bytes) outweighs the reduction in backup BF size. n_features = 4096 sits at the crossover where oracle quality and model memory are best balanced for this dataset and feature type.

honest discussion

Limitations

  • L1
    FNR = 0 is a split property, not a model result. System FNR is zero because all good URLs appear in the training set. The backup BF memorises every member the model missed at training time, and since those same members appear at test time, they are always recovered. This is not a claim about model quality — a weaker model simply pushes more members into the backup BF, inflating its size.
  • L2
    All latency figures are amortised batch throughput costs, not single-query latency. Every number — Python and C++ alike — is measured via time.perf_counter_ns() in Python, wrapping a single batched pybind11 call with 3 untimed warmup runs and a mean over 10 timed trials, divided by batch size. This reflects steady-state throughput under batch load. A real serving system processing one URL at a time would experience higher latency on both sides due to per-call Python↔C++ crossing overhead that is amortised away in the batch measurement.
  • L3
    Memory accounting is asymmetric. Reported memory includes only the Logistic Regression model weights and the backup Bloom filter. Python runtime overhead, temporary data structures, and scikit-learn internal objects were not included, so the comparison focuses on the memory used by the learned Bloom filter itself rather than language-specific overhead.
